# Customer-side IAM role (CloudFormation)

> P1 item 2: deployable stack the prospect runs in **their own AWS
> account** to grant Constat read-only access. Replaces the
> pre-pilot "doc note" with a one-click CloudFormation stack.

## What this is

The Constat SaaS scans your RDS fleet to find Extended Support
exposure (and other facts in V2). To do that, it needs a role in
your account that it can `sts:AssumeRole` into. This CloudFormation
template creates that role with the **minimum permissions** the
SaaS needs, bound to:

- a specific SaaS account (via the trust policy's `Principal.AWS`),
- a shared `ExternalId` (the confused-deputy mitigation — F-06).

The template lives at `infra/customer-iam-role.yaml` in the Constat
repo. It is **not** deployed by the Constant team. It is deployed
by **your** AWS admin in **your** account.

## What it creates

```
ConstatCollectorRole       AWS::IAM::Role
├── Trust policy:           SaaS account + ExternalId condition
├── Inline policy:
│   ├── rds:Describe*        (read-only RDS inventory)
│   ├── iam:GetRole          (the SaaS verifies its own role)
│   └── sts:TagSession       (Self-Tag for CloudTrail attribution)
└── Tags:
    ├── ManagedBy:           constat-collector-stack
    └── CostCenter:          (your value, optional)
```

**No write permissions.** The SaaS cannot create, modify, or delete
RDS instances, IAM roles, or any other resource. It can only read.

**No `iam:PassRole`.** Confused-deputy mitigation. The SaaS cannot
use your role to escalate to other roles.

## Before you deploy

The Constat team gives you two values during onboarding:

| Parameter | What to put | Where it comes from |
|---|---|---|
| `ConstatAccountId` | The SaaS's AWS account ID (12 digits) | Constat onboarding email |
| `ExternalId` | A 32+ char random string | Generated by the Constat team, shared over a secure channel |

Keep the `ExternalId` in your password manager. The SaaS will store
the same value in Secrets Manager. If a third party learns your
`ExternalId`, they can assume this role from any compromised
identity in the SaaS account — rotate it via Secrets Manager on
both sides if you suspect a leak.

## Deploy

### Option A — AWS Console

1. Sign in to the AWS account you want Constat to scan.
2. Go to CloudFormation → Create stack → With new resources.
3. Upload `infra/customer-iam-role.yaml`.
4. Fill in the parameters. Use the values the Constat team sent you.
5. Review the IAM policy (the template is short — read it).
6. Create the stack. Wait for `CREATE_COMPLETE` (~30s).
7. Go to Outputs. Copy `RoleArn`. Send it back to the Constat team
   (over the same secure channel they used for the ExternalId).

### Option B — AWS CLI

```bash
aws cloudformation create-stack \
  --stack-name constat-collector \
  --template-body file://infra/customer-iam-role.yaml \
  --parameters \
      ParameterKey=ConstatAccountId,ParameterValue=<12-digit-id> \
      ParameterKey=ExternalId,ParameterValue=<32+char-secret> \
      ParameterKey=TagsCostCenter,ParameterValue=<your-cost-center> \
  --capabilities CAPABILITY_NAMED_IAM \
  --region eu-west-1   # or whichever region you prefer; the role is global
```

Get the RoleArn:

```bash
aws cloudformation describe-stacks \
  --stack-name constat-collector \
  --query 'Stacks[0].Outputs[?OutputKey==`RoleArn`].OutputValue' \
  --output text
```

Send it to Constat.

### Option C — Terraform

If you manage your AWS with Terraform, import the stack:

```hcl
resource "aws_cloudformation_stack" "constat_collector" {
  name         = "constat-collector"
  template_url = "https://raw.githubusercontent.com/<constat-repo>/main/infra/customer-iam-role.yaml"
  # Or template_body = file("${path.module}/customer-iam-role.yaml")

  parameters = {
    ConstatAccountId = var.constat_account_id
    ExternalId       = var.constat_external_id
    TagsCostCenter   = var.cost_center
  }

  capabilities = ["CAPABILITY_NAMED_IAM"]
}
```

Store `var.constat_external_id` in your secret manager
(`aws_ssm_parameter` with `type = "SecureString"`, or AWS Secrets
Manager). Do **not** commit it to git.

## What happens after you share the RoleArn

1. The Constat team stores the role ARN + matching `ExternalId` in
   their Secrets Manager.
2. The SaaS collector calls `rds:DescribeDBInstances` in each
   region, once per day (or on demand via `POST /collect/aws`).
3. The collector writes the inventory to Constat. Insights appear
   on the next `/insights` page refresh.
4. Every action the SaaS takes is tagged with its request_id
   (see `apps/api/src/constat_api/middleware.py`) and visible in
   your CloudTrail under the role's session name
   (`constat-<uuid>`).

## What you can audit

```bash
# All actions taken by the Constat role in the last 7 days.
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=Username,AttributeValue=constat-* \
  --start-time "$(date -u -d '7 days ago' '+%Y-%m-%dT%H:%M:%S')" \
  --max-items 50
```

(Adjust the date format for your platform.) Every event is
`EventName=DescribeDBInstances` or similar — read-only, tagged
with the SaaS's request_id.

## How to rotate the ExternalId

If the ExternalId is suspected to be leaked:

1. Generate a new 32+ char secret in your password manager.
2. Send it to the Constat team (secure channel).
3. Constat updates their Secrets Manager; the next SaaS scan uses
   the new id.
4. Update the stack parameter (`ExternalId`) and apply.
5. The role's `Condition.sts:ExternalId` is now the new value. The
   old one is dead.

There is no downtime — the SaaS retries on a transient
`InvalidClientTokenId` / `SignatureDoesNotMatch`.

## How to revoke access

```bash
# Option 1: delete the stack
aws cloudformation delete-stack --stack-name constat-collector

# Option 2: detach the policy (role remains, but does nothing useful)
aws iam delete-role-policy \
  --role-name constat-collector \
  --policy-name constat-collector-readonly
```

Either stops the SaaS scan within minutes. CloudTrail will show
the `DeleteStack` or `DeleteRolePolicy` event.

## Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| Stack fails with `AccessDenied` on `iam:CreateRole` | Your IAM user/role doesn't have permission to create IAM roles | Use an admin user, or grant `iam:CreateRole` + `iam:PutRolePolicy` |
| Stack fails with `InvalidParameter: ExternalId` | The ExternalId has invalid characters (allowed: alphanumeric + `=+/.,@-_`) | Re-generate, ensure no spaces or non-ASCII |
| Constat scan returns `AccessDenied: User ... is not authorized` | The SaaS's `Principal.AWS` in the trust policy doesn't match the SaaS's actual account | Re-check `ConstatAccountId` with the Constat team |
| `SignatureDoesNotMatch` on the SaaS side | ExternalId mismatch between your stack and the SaaS's Secrets Manager | Coordinate the value with the Constat team; rotate if needed |
| CloudTrail shows the Constat role but no `TagSession` events | The SaaS's first scan may not have tagged the session. Normal. | Wait for the next scan. |

## What is NOT in V1

- **No `aws:SourceIp` condition.** The SaaS runs in ECS Fargate
  without predictable IPs. When the SaaS is behind a known NAT,
  add an `aws:SourceIp` condition to the trust policy. Out of
  scope here.
- **No per-region scoping.** The role is global; the SaaS scans
  every region you have enabled. If you want to limit to one
  region, fork this template and add an `aws:RequestedRegion`
  condition. The Constat team can also configure the SaaS-side
  targets to skip regions.
- **No time-window condition.** The SaaS can scan any time.
  Add an `aws:CurrentTime` condition if you want business-hours
  only (the Constat team should know about this — it changes the
  SLO for their insight freshness).
- **No multi-account.** V1 is single-account SaaS. If you have
  AWS Organizations and want Constat to scan member accounts,
  fork this and use `aws:PrincipalOrgID` or stack-set deployment.

## What the Constat team owns (not you)

- The SaaS-side task role (`apps/infra/iam.tf::aws_iam_role.task`).
  This is in the SaaS's account, not yours.
- The Secret Manager storage of your ExternalId.
- The scan schedule and the rule definitions.

If you change your side (rotate ExternalId, delete the stack,
update tags), the Constant team will see the next scan fail and
follow up. The SaaS is idempotent on transient errors; permanent
errors surface in `/status` and the operator's UI.
